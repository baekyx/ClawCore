# ClawCore — 多层记忆 + Skill 自进化的通用 Agent 助手

**技术选型**：Python | DeepSeek | Postgres + pgvector | BGE-M3 | BM25 + jieba | Docker | SQLite | HelloAgents

GitHub: https://github.com/baekyx/ClawCore | 3341行 / 36模块

---

## 一、Agent Loop — ReAct 循环引擎

### 选型意图

| 决策 | 选了 | 为什么 | 没选 |
|------|------|--------|------|
| **Agent 范式** | ReAct (Function Calling) | 主流最稳定，LLM 原生支持 tool_choice，解析成功率 99%+ | ReWOO 省 Token 但灵活性差 |
| **LLM** | DeepSeek | 便宜、中文好、兼容 OpenAI 格式  | GPT-4o 贵 10 倍 |
| **LLM 适配** | 复用 HelloAgents 三种适配器 | 已学框架，直接覆盖 OpenAI/Anthropic/Gemini | 自己写适配层重复造轮子 |
| **并行策略** | asyncio.gather + Semaphore(3) | 内置工具串行保逻辑，业务工具并行减延迟 | 全部串行太慢，全并行危险 |
| **容错** | CircuitBreaker 熔断器 | 连续失败 3 次自动熔断 300 秒，防止坏工具耗尽 API 费用 | 无限重试会烧钱 |
| **降级** | 构造函数可选参数 | Postgres 不可用时 Agent 照常运行，只是不记跨会话信息 | 强依赖会崩溃 |

### STAR

- **S**：LangChain 黑盒重，不适合深度定制
- **T**：手写 ReAct 循环，不依赖第三方 Agent 框架
- **A**：手写 while 主循环（推理→工具调用→观察→压缩→循环），9 个工具并行调度
- **R**：平均 1-2 步，2-3 秒响应，降级模式正常运行

## 二、三层记忆架构

### 选型意图

| 决策 | 选了 | 为什么 | 没选 |
|------|------|--------|------|
| **L1 会话态** | SQLite | 零配置、嵌入式、单文件，会话数据生命周期短不需要分布式 | Redis 需额外运维 |
| **L2 工作记忆** | Postgres JSONB | 灵活 Schema、Upsert 防重复、支持全文搜索 | MongoDB 查询语法不自然 |
| **L3 长期记忆** | Postgres pgvector | 向量检索+事务一致，一套数据库搞定L2+L3 | Qdrant/Milvus 需单独部署 |
| **稠密检索** | BGE-M3 (1024维) | 中文 SOTA，支持 Dense+Sparse 双表征，本地免费 | OpenAI text-embedding-3 按量付费且数据出境 |
| **稀疏检索** | BM25 + jieba | 精确关键词匹配，无 GPU，与 Dense 互补 | 纯 Dense 丢精确匹配 |
| **融合算法** | RRF (k=60) | 无需调参，双路分数直接归一化 | 学习融合需要大量标注数据 |
| **容器化** | Docker pgvector/pg16 | 一行命令起数据库，和代码一起版本管理 | 手动装 Postgres 配置繁琐 |

### STAR

- **S**：LLM 上下文窗口有限，每次新会话 Agent 都"失忆"
- **T**：设计多层记忆，跨会话保留信息，支持语义+关键词混合检索
- **A**：L1 SQLite(即时读写) + L2 Postgres JSONB(用户画像 3 命名空间 Upsert 语义) + L3 pgvector+BGE-M3+BM25+RRF(混合检索)；记忆沉淀管道 L1→L2→L3 自动运行
- **R**：实测"记住张三/北京"→新对话正确回忆；HitRate@10 99.4%，MRR@10 0.67

## 三、四层上下文压缩

### 选型意图

| 决策 | 选了 | 为什么 | 没选 |
|------|------|--------|------|
| **L1** | Token 阈值截断 | O(n) 零成本，永远最先触发 | 不截断直接爆窗口 |
| **L2** | 规则引擎去重+合并 | O(n) 零成本，MD5+Jaccard 纯 Python | 调 LLM 去重又慢又费钱 |
| **L3** | 轻量 LLM 结构化摘要 | 只在 L1+L2 不够时触发，五段式保留关键信息 | 全量 LLM 摘要成本高、只截断精度低 |
| **L4** | 自适应阈值 | 长对话更激进、短对话更保守，避免过度/不足压缩 | 固定阈值一刀切 |
| **顺序设计** | L1→L2→L3 递进 | 由快到慢、由轻到重，前面的解决大部分问题，后面的兜底 | 跳过 L1/L2 直接调 LLM 又慢又贵 |

### STAR

- **S**：复杂任务数百条消息会超出 LLM 上下文窗口
- **T**：多层压缩，由快到慢递进，信息损失最小化，体积降 30%
- **A**：L1 Budget 截断(80%触发)→L2 MD5 去重+Jaccard 合并→L3 五段式摘要(LLM)→L4 动态参数(轮次/工具数/消息长度)
- **R**：体积-30%，200 轮长会话正确率 77%

## 四、Skill 自进化

### 选型意图

| 决策 | 选了 | 为什么 | 没选 |
|------|------|--------|------|
| **加载策略** | 渐进式目录披露 | 启动只加载元数据(~100tokens/skill)，按需加载 body(~2000tokens) | 全量注入浪费上下文 |
| **前向循环** | System Prompt 引导 | Agent 被动感知可用技能，调用 `skill_invoke` 工具加载 | 自动注入所有技能撑爆 prompt |
| **后向循环** | 离线 Review | 解耦在线/离线，不影响主循环性能 | 在线学习增加延迟且不稳定 |
| **版本管理** | 语义版本 + 相似度判定 | SequenceMatcher 对比新旧 body，<50%→MAJOR，>3行新增→MINOR | 人工判定费时、无版本管理丢失历史 |
| **验证** | 自动化校验 | 格式/工具引用/版本号三项自动检查 | 不验证的 Skill 可能误导 Agent |

### STAR

- **S**：传统 Agent 能力写死在 Prompt 里，成功经验无法复用
- **T**：让 Agent 自动提取、验证、沉淀新技能，越用越聪明
- **A**：前向循环(渐进式披露+按需加载) + 后向循环(高频工具链≥3次→LLM生成描述→自动校验→语义版本 bump→存档回滚) + 成功率追踪(低分自动降权)
- **R**：100 次对话自动沉淀 ≥3 个可验证 Skill，引用成功率 >80%，支持热重载
